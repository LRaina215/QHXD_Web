### 当前目标

### 当前结论

项目最终采用 **NUC11 + RK3588 + RT-Thread** 三节点协同：

- NUC11：主智能计算平台
- RT-Thread：底层实时执行与安全闭环
- RK3588：业务交互与边缘服务节点

当前 NUC11 正在推进 P0 的导航、SLAM 与点到点能力，因此 RK3588 第一阶段不承担主导航任务，而是优先建立中台能力。

## 6. 已知问题与风险

### 风险 1：RK3588 需求容易膨胀

如果在第一阶段就接入语音、大模型、图传、真实多机通信，极易拖慢 P0。必须坚持先做空壳中台，再逐步接真实链路。该策略符合 Vibe Coding 的 MVP 小步迭代原则。fileciteturn3file0

### 风险 2：页面与字段不先锁定，后续会反复返工

必须先固定页面清单、数据模型和 API 草案，否则后续 AI 容易改 UI、改字段、改逻辑。fileciteturn3file0

### 风险 3：真实设备字段不一致

NUC11 与 RT-Thread 后续真实字段可能和当前文档不完全一致，因此需要采用“内部统一模型 + 外部适配层”的方式。

### 风险 4：AI 一次生成过大代码块

必须强制按最小切片开发，每个切片通过后立即 commit，避免 AI 一次写跨模块大段代码。fileciteturn3file0

## 7. 开发纪律

### 7.1 小步迭代

严格按最小可验证切片推进：

1. 文档
2. 后端空壳
3. WebSocket
4. 模拟数据
5. 前端 Dashboard
6. 其他页面

每完成一小步，立即提交 Git。fileciteturn3file0

### 7.2 Prompt 防护

后续让 AI 修改代码时，统一附加：

> 仅修改我指定的文件和代码范围。禁止顺手重构其他代码，禁止修改 UI 风格，禁止更改无关逻辑。

### 7.3 两跳阻断

若 AI 连续两次修复失败或引入更多错误，立即停止自动修复，改为人工最小复现 + 加日志 + 再喂精确上下文。fileciteturn3file0
