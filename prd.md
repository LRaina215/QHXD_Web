# PRD：RK3588 车载交互与状态中台（第一阶段）

## 1. 文档目的

本文档用于明确 **RK3588 车载交互与状态中台** 在项目第一阶段的产品需求、功能边界、页面范围与验收标准，并以当前已经通过验收的 Phase 1 实现为基准同步更新。

---

## 2. 项目背景

本项目采用三节点协同架构：

- **NUC11**：主智能计算平台，负责 SLAM、定位、路径规划、主视觉感知、任务主状态机等
- **RT-Thread 控制器**：底层实时执行与安全闭环，负责电机控制、编码器 / IMU 采集、姿态稳定、急停与失联保护
- **RK3588**：车载业务交互与边缘服务节点，负责状态聚合、任务入口、Web 服务、语音交互预留、巡检数据服务等

当前 NUC11 正在推进 P0 任务，因此 RK3588 第一阶段必须 **独立推进、不得阻塞主导航闭环**。

---

## 3. 产品定位

RK3588 在第一阶段不承担主导航与主控制任务，而是作为：

1. **任务入口节点**：接收 Web / App / 后续语音输入产生的任务请求
2. **状态中台节点**：聚合 NUC11 与 RT-Thread 的状态信息，形成统一状态快照
3. **数据服务节点**：缓存机器人状态、记录告警与任务日志
4. **Web 可视化节点**：向前端提供状态展示与基础命令接口

---

## 4. 当前版本目标（V1）

### 4.1 目标

在 RK3588 上建立一个可运行的后端 + 前端空壳系统，使其具备以下能力：

- 启动 API 服务与 WebSocket 服务
- 定义统一数据模型
- 使用模拟数据驱动页面
- 提供状态查询与任务命令接口
- 优先交付单页 Dashboard 验收版，覆盖状态展示、告警展示与基础命令入口
- 为后续与 NUC11、RT-Thread 真机对接预留稳定接口

### 4.2 非目标

当前版本不实现以下内容：

- 真实 NUC11 与 RT-Thread 通信联调
- 真实地图展示与图传
- 主导航、SLAM、YOLO 推理
- 完整权限系统与公网部署
- 完整语音识别与大模型推理

## 6. 核心场景

### 场景 A：开发阶段状态调试

开发成员需要在 RK3588 页面查看当前位姿、任务状态、电量、急停、传感器数据等，以验证系统集成效果。

### 场景 B：演示阶段任务下发

演示操作者通过页面选择目标点并下发任务，系统记录命令并更新页面状态。

### 场景 C：异常与巡检展示

开发 / 演示人员查看最近告警、环境传感器数据与任务日志，展示机器人具备巡检与状态上报能力。

## 7. 功能需求与验收标准

### F1. 后端服务启动

**说明**：RK3588 能启动基础后端服务，对外暴露 HTTP 与 WebSocket 接口。

**验收标准**：

- 启动命令执行后，服务进程正常运行
- `GET /health` 返回 HTTP 200
- WebSocket 连接建立成功
- 启动失败时日志中输出明确错误原因
- 未加载设备数据时系统仍可进入模拟模式

### F2. 统一数据模型

**说明**：系统内对机器人状态与任务命令使用统一结构，避免不同模块字段含义不一致。

**验收标准**：

- 至少定义以下结构：
  - `robot_pose`
  - `nav_status`
  - `task_status`
  - `device_status`
  - `env_sensor`
  - `alert_event`
- 字段语义统一，不出现重复命名
- 前后端与文档口径一致

### F3. 状态缓存与查询

**说明**：RK3588 能缓存最新状态快照，并通过 API 查询。

**验收标准**：

- 可通过接口查询最新机器人状态
- 最近告警列表可查询
- 当前任务可查询
- 所有状态更新时间可追踪

### F4. WebSocket 实时推送

**说明**：后端能将状态变化实时推送给前端。

**验收标准**：

- 前端连接后可收到状态更新
- 状态变化后 1 秒内页面有可见更新
- 连接断开时页面显示离线
- 重连后可恢复订阅
- 无真实设备时模拟数据可驱动页面更新

### F5. 任务命令接口

**说明**：前端可向 RK3588 下发基础命令，由 RK3588 进行校验、记录并转发给后续业务层。

**当前支持命令**：

- `go_to_waypoint`
- `start_patrol`
- `pause`
- `resume`
- `return_home`

**验收标准**：

- 命令缺少参数时返回结构化错误信息
- 不支持的命令不可入队
- 成功命令写入命令日志
- 所有命令记录来源、时间与参数摘要
- 模拟模式下命令可触发页面状态变化

### F6. 基础 Web 页面

**说明**：提供一个可运行的前端框架用于展示状态和发送指令。

**当前 Phase 1 验收形态**：

- 采用单页 Dashboard 作为最小交付页面
- Dashboard 内包含状态展示、告警展示和基础命令入口
- 独立 Mission / Devices / Logs 页面延后到下一阶段补强

**验收标准**：

- Dashboard 至少展示：在线状态、当前任务、目标点、电量、急停、环境传感器
- Dashboard 内至少可发送基础任务命令
- Dashboard 内至少展示最近告警和命令执行结果
- 页面在无真实设备时也可访问

### F7. 日志与本地持久化

**说明**：使用 SQLite 保存中台级别的关键记录。

**保存范围**：

- 命令日志
- 最近告警
- 状态摘要 / 状态快照

**验收标准**：

- SQLite 表结构有明确设计
- 任务命令日志可查询
- 最近告警可查询
- 程序重启后历史日志仍可读取

## 9. 页面与交互原则

### 9.1 视觉基调

- 工业设备监控风格
- 深色背景或高对比卡片式布局
- 单页看板式布局

### 9.2 交互原则

- 首页优先展示系统是否在线
- 任务操作必须有明显反馈
- 错误提示清晰，不使用模糊文案
- 页面必须支持模拟模式

---

## 10. 用户流程

### 流程 U1：查看机器人当前状态

用户打开 Dashboard → 页面连接 WebSocket → 获取最新状态快照 → 实时刷新状态卡片。

### 流程 U2：下发目标点任务

用户进入 Dashboard → 输入目标点 → 点击下发 → 后端记录日志 → 页面显示任务进入执行中。

### 流程 U3：查看最近异常

用户进入 Dashboard 告警区 → 查看最近告警列表 → 后续扩展时再进入独立日志页查看详情。

---

## 11. 风险与约束

### 风险

- NUC11 与 RT-Thread 的真实字段尚未完全固定
- 第一阶段若过早接入真实链路，可能拖慢 P0 主导航推进
- 页面需求若不先锁定，后续 AI 容易擅自改变 UI 或结构

### 约束

- 第一阶段必须独立于主导航开发推进
- 禁止把 RK3588 变成主导航或主控制平台
- 所有功能先用模拟数据跑通，再逐步替换为真实数据

---

## 12. 第一阶段完成定义（Definition of Done）

当满足以下条件时，认为第一阶段产品需求达成：

1. `prd.md`、`arc.md`、`project.md` 三份文档完成并入库
2. RK3588 后端服务可运行
3. 至少一个 WebSocket 状态推送链路可工作
4. 前端单页 Dashboard 可访问并展示核心状态
5. 页面可显示模拟状态数据
6. 基础命令接口可用并写入日志
7. 页面风格与主看板布局被锁定，不再在后续开发中随意漂移
