# prd_p2.md

## 1. 文档定位

本文件定义 **Phase 2：RK3588 ↔ NUC 真实桥接阶段** 的产品范围、功能目标、页面需求与验收标准。

本阶段建立在 Phase 1 已完成并通过验收的基础上。Phase 1 已实现：

- RK3588 后端服务骨架
- WebSocket 状态推送骨架
- SQLite 本地持久化
- 前端 Dashboard 基础页面
- mock 状态链路与任务接口占位

Phase 2 的目标不是新增大量炫技功能，而是将 RK3588 从“空壳中台”升级为**真实接入 NUC 的任务与状态中台**。这一阶段仍属于整体项目的 P0 延伸部分，优先服务于“导航—配送任务—状态显示”的基础闭环。fileciteturn8file0turn7file0turn3file0

## 2. 阶段目标

### 2.1 总目标

实现 RK3588 与 NUC11 之间的真实状态与任务桥接，使 RK3588 具备以下能力：

1. 接收 NUC 的真实导航与任务状态
2. 将真实状态推送给 Web 前端
3. 接收前端任务命令并转发给 NUC
4. 接收 NUC 返回的执行结果并更新页面状态
5. 支持 mock / real 双模式切换，保证开发与联调均可进行

### 2.2 本阶段不做的内容

本阶段**不实现**以下功能：

- RK3588 主导航、主 SLAM、主视觉推理
- RT-Thread 与 RK3588 的主控制直连替代 NUC
- 真实语音识别链路
- OpenClaw 集成
- 真实视频图传与 WebRTC 服务
- 新环境传感器硬件接入
- 视觉异常检测联动

这些功能属于后续 P1 / P2 范围，不纳入本阶段验收。fileciteturn8file0turn7file0

## 3. 用户与使用场景

### 3.1 目标用户

- 机器人开发成员
- 比赛现场操作者
- 负责联调的上位机 / 中台开发成员

### 3.2 核心场景

#### 场景 A：查看真实机器人状态

操作者打开 RK3588 Web 页面，实时查看当前位姿、导航状态、目标点、任务状态、电量和告警信息。

#### 场景 B：通过中台下发基础任务

操作者在 Web 页面选择目标点，点击发送任务，RK3588 将结构化命令转发给 NUC，NUC 返回任务接收结果并开始执行。

#### 场景 C：任务过程状态跟踪

任务执行过程中，RK3588 持续接收 NUC 状态并推送给前端，页面同步更新为 running / paused / succeeded / failed 等状态。

#### 场景 D：联调阶段回退到 mock 模式

当 NUC 侧不稳定或联调未准备就绪时，RK3588 可切换到 mock 模式，继续完成前端页面与状态逻辑调试。

## 4. 功能范围与验收标准

### F1：真实状态接入

**说明**：RK3588 后端接收来自 NUC 的真实状态数据，替代 Phase 1 的纯 mock 数据源。

**输入状态至少包括：**

- `robot_pose`
- `nav_status`
- `current_goal`
- `task_status`
- `detection_status`（可占位）

**验收标准：**

- RK3588 可接收并缓存 NUC 的真实状态
- 前端刷新后可看到非 mock 的实时状态
- 状态更新延迟在本地局域网联调条件下可接受
- NUC 断开后页面进入离线态，并给出明确提示
- 重连后状态恢复正常

---

### F2：任务命令桥接

**说明**：RK3588 接收前端任务命令，并将其转发给 NUC 的任务管理模块。

**本阶段支持命令：**

- `go_to_waypoint`
- `start_patrol`
- `pause`
- `resume`
- `return_home`

**验收标准：**

- 前端点击按钮后，RK3588 可生成结构化命令
- NUC 能收到至少 3 个真实命令
- NUC 对命令有明确接收结果返回
- RK3588 能记录命令日志
- 前端可以看到命令后的状态变化

---

### F3：状态—命令—反馈闭环

**说明**：命令下发后，RK3588 不仅要“发出去”，还要能看到任务状态的真实变化。

**验收标准：**

- 下发任务后，任务状态从 idle 切换为 running
- 暂停命令后，任务状态切换为 paused
- 恢复命令后，任务状态恢复 running
- 返回起点命令后，任务目标更新并可见
- 所有状态变化在前端页面中可观察到

---

### F4：mock / real 双模式切换

**说明**：系统保留 mock 模式，并新增 real 模式，用于开发、联调和回退。

**验收标准：**

- 启动配置中可明确指定 mock 或 real 模式
- mock 模式下系统仍可独立演示
- real 模式下系统使用 NUC 真实状态
- 模式切换后页面与后端行为符合预期
- 切换失败时有明确报错

---

### F4.1：契约冻结补充

**说明**：在真实通信正式接入前，先冻结 RK3588 ↔ NUC 的公开契约，保证前后端、RK3588 与 NUC 对接口径一致。

**本轮冻结字段：**

- `robot_pose`
- `nav_status`
- `task_status`
- `device_status`
- `env_sensor`
- `alert_event`
- `system_mode`

**本轮冻结接口：**

- `GET /api/state/latest`
- `GET /api/alerts`
- `GET /api/tasks/current`
- `POST /api/mission/go_to_waypoint`
- `POST /api/mission/start_patrol`
- `POST /api/mission/pause`
- `POST /api/mission/resume`
- `POST /api/mission/return_home`
- `POST /api/system/mode/switch`

**验收标准：**

- 契约在代码中有显式 typed schema
- 请求 / 响应模型明确且前后一致
- 在未接入真实 NUC 时，接口仍可返回结构化 JSON
- mock / real 模式切换接口已保留，但当前不要求真实切换数据源

---

### F5：日志与可追踪性增强

**说明**：在真实联调阶段补充命令日志、状态接收日志与错误日志，便于定位故障位置。

**验收标准：**

- 每次任务命令有时间戳、来源、参数和结果记录
- 每次状态接入异常可定位是连接问题、字段问题还是解析问题
- SQLite 中至少能查到最近命令日志和最近告警
- 页面或接口可查询最近告警 / 最近任务状态摘要

---

### F6：前端 Phase 2 显示增强

**说明**：前端在不大改 UI 的前提下，从“展示 mock 数据”升级为“展示真实状态并反馈联调信息”。

**页面至少展示：**

- 在线 / 离线状态
- 当前位姿
- 当前任务
- 当前目标点
- 当前导航状态
- 电量
- 急停状态
- 最近告警
- 当前模式（mock / real）

**验收标准：**

- 页面能明显区分 mock 与 real 模式
- 真实状态变化时前端自动刷新
- 连接断开时页面提示清楚
- 任务命令按钮与真实后端联动正常

## 5. 页面清单与页面要求

### 5.1 Dashboard（保留，升级）

**目标**：作为真实状态总览页。

**必显内容：**

- 在线状态
- 机器人位姿
- 任务状态
- 当前目标点
- 电量与急停
- 模式标识（mock / real）
- 最近告警

**验收重点：**

- 看一眼能判断机器人是否在线、是否在执行、是否有异常

---

### 5.2 Mission（保留，升级）

**目标**：完成任务下发与命令反馈。

**必备操作：**

- 选择 waypoint
- 下发目标点
- 开始巡检
- 暂停
- 恢复
- 返航

**验收重点：**

- 操作后能看到真实任务状态变化，不是静态按钮回显

---

### 5.3 Logs（保留，增强）

**目标**：查看命令记录与状态异常。

**必显内容：**

- 最近命令日志
- 最近告警日志
- 关键错误信息

**验收重点：**

- 出问题时能快速定位是命令没发出、NUC 没响应，还是状态没回传

---

### 5.4 Devices / Debug（可合并）

**目标**：联调阶段查看中台连接情况。

**必显内容：**

- NUC 连接状态
- RT-Thread 状态占位（此阶段可先显示缓存值）
- 当前模式
- 最近一次状态更新时间

## 7. Phase 2 验收通过标准

本阶段通过需同时满足以下条件：

1. RK3588 能接收到 NUC 的真实状态
2. 前端能显示真实状态并实时更新
3. 至少 3 个命令可真实桥接到 NUC
4. 任务状态能形成“命令—执行—反馈”闭环
5. mock / real 模式可切换
6. 日志可追踪、可定位基本问题

## 8. Phase 2 不通过判定

出现以下任一情况，判定本阶段不通过：

- 前端仍只能展示 mock 数据
- 命令接口只有表面返回，没有真实桥接到 NUC
- 状态变化没有闭环反馈
- 联调出现问题时无法判断故障位置
- 文档和代码严重不一致
